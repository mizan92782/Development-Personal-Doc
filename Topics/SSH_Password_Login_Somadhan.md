# SSH দিয়ে Username, IP ও Password এ লগইন করতে না পারার সমস্যা ও সমাধান

## ভূমিকা

একটি VPS/Cloud সার্ভারে (Ubuntu 24.04) নিজের কম্পিউটার থেকে বিনা পাসওয়ার্ডে SSH লগইন করা যাচ্ছিল, কিন্তু টিমের অন্য মেম্বার তার কম্পিউটার থেকে **username + IP + password** দিয়ে লগইন করতে পারছিল না। এই ডকুমেন্টে সমস্যাটি কেন হয়েছিল, কোন কোন ফাইল/ফোল্ডার এর সাথে জড়িত এবং কীভাবে সমাধান করা হয়েছে তা ধাপে ধাপে ব্যাখ্যা করা হলো।

---

## সমস্যাটি কেন হয়েছিল

মূলত দুইটি কারণে এই সমস্যা তৈরি হয়েছিল:

### ১. সার্ভারে Password Authentication বন্ধ ছিল

Ubuntu/Linux সার্ভারে SSH এর কনফিগারেশন থাকে এই ফাইলে:

```
/etc/ssh/sshd_config
```

এই ফাইলে একটি লাইন থাকে:

```
#PasswordAuthentication yes
```

শুরুতে দেখা যায় লাইনের আগে `#` (হ্যাশ চিহ্ন) দেওয়া ছিল, যার মানে এই সেটিংসটি **কমেন্ট করা** ছিল অর্থাৎ নিষ্ক্রিয় ছিল। ফলে সার্ভার কোনো ব্যবহারকারীকে পাসওয়ার্ড দিয়ে লগইন করার অনুমতিই দিচ্ছিল না — শুধুমাত্র SSH Key দিয়ে লগইন করা যেত।

### ২. Root ইউজারের কোনো Password সেট করা ছিল না

এটাই ছিল প্রকৃত সমস্যা। ক্লাউড/VPS সার্ভার (যেমন DigitalOcean, Vultr, Hetzner ইত্যাদি) সাধারণত সিকিউরিটির স্বার্থে root ইউজারের জন্য কোনো password সেট করে না — শুধু SSH key দিয়ে অ্যাক্সেস দেওয়ার ব্যবস্থা রাখা হয়। তাই root এর password যদি সেট না থাকে, তাহলে `PasswordAuthentication yes` করলেও কেউ password দিয়ে লগইন করতে পারবে না, কারণ যাচাই করার মতো কোনো password-ই নেই।

### ৩. নিজের কম্পিউটারে Key দিয়ে অটো-লগইন হচ্ছিল কেন

নিজের কম্পিউটারে SSH কনফিগারেশন ফাইল ছিল:

```
~/.ssh/config
```

এই ফাইলে `outdora` নামে একটি Host এন্ট্রি ছিল, যেমন:

```
Host outdora
    HostName 187.124.228.249
    User root
    IdentityFile ~/.ssh/outdoora
    IdentitiesOnly yes
```

এখানে `IdentityFile ~/.ssh/outdoora` অংশটি একটি প্রাইভেট SSH key নির্দেশ করে, যেটির পাবলিক key আগে থেকেই সার্ভারের এই ফাইলে যুক্ত ছিল:

```
/root/.ssh/authorized_keys
```

যেহেতু নিজের কম্পিউটারের private key এর সাথে সার্ভারের `authorized_keys` ফাইলের public key মিলে যাচ্ছিল, তাই SSH কোনো পাসওয়ার্ড না চেয়েই সরাসরি লগইন করিয়ে দিচ্ছিল। কিন্তু টিমমেটের কম্পিউটারে এই key না থাকায় তাকে password ব্যবহার করতে হচ্ছিল, আর সেটাই কাজ করছিল না উপরের দুইটি কারণে।

## কোন কোন ফোল্ডার/ফাইল জড়িত ছিল

| ফাইল/ফোল্ডার | অবস্থান | কাজ |
|---|---|---|
| SSH সার্ভার কনফিগারেশন | `/etc/ssh/sshd_config` | সার্ভার সাইডে password/key authentication চালু-বন্ধ নিয়ন্ত্রণ করে |
| Authorized keys | `/root/.ssh/authorized_keys` | কোন কোন public key দিয়ে বিনা পাসওয়ার্ডে লগইন করা যাবে তা রাখে |
| লোকাল SSH কনফিগ | `~/.ssh/config` | নিজের কম্পিউটারে কোন Host এর জন্য কোন key/user ব্যবহার হবে তা নির্ধারণ করে |
| প্রাইভেট Key | `~/.ssh/outdoora` (নিজের কম্পিউটারে) | লগইনের সময় ব্যবহৃত গোপন key |

## সমাধানের ধাপসমূহ

### ধাপ ১: Password Authentication চালু করা

সার্ভারে SSH করে `sshd_config` ফাইলে গিয়ে নিচের কমান্ড দিয়ে সেটিং পরিবর্তন করা হয়েছে:

```bash
sudo sed -i 's/#PasswordAuthentication yes/PasswordAuthentication yes/' /etc/ssh/sshd_config
```

### ধাপ ২: SSH সার্ভিস রিস্টার্ট করা

পরিবর্তন কার্যকর করতে SSH সার্ভিস রিস্টার্ট করতে হয়:

```bash
sudo systemctl restart ssh
```

> **লক্ষণীয়:** কিছু সার্ভারে সার্ভিসের নাম `sshd`, আবার কিছু সার্ভারে (বিশেষত Ubuntu) `ssh`। ভুল নাম দিলে "Unit not found" এরর আসতে পারে।

### ধাপ ৩: Root ইউজারের জন্য Password সেট করা

এটিই ছিল আসল সমাধান — root এর জন্য একটি password সেট করে দেওয়া হয়:

```bash
sudo passwd root
```

অথবা স্ক্রিপ্টের মাধ্যমে:

```bash
echo 'root:YourStrongPassword' | sudo chpasswd
```

### ধাপ ৪: টেস্ট করা

টিমমেট যেভাবে লগইন করবে (শুধু password দিয়ে, কোনো key ছাড়া) সেভাবে টেস্ট করা হয়েছে:

```bash
ssh -o PubkeyAuthentication=no root@187.124.228.249
```

এবং পাসওয়ার্ড চাইলে সঠিক password দিলে লগইন সফল হয়েছে।

---

## টিমমেট কীভাবে লগইন করবে

টিমমেট তার কম্পিউটার থেকে সাধারণভাবে নিচের কমান্ড দিয়ে লগইন করতে পারবে:

```bash
ssh root@187.124.228.249
```

Password চাইলে নির্ধারিত password দিতে হবে।

---

## নিরাপত্তা সংক্রান্ত পরামর্শ

- **Root এর password দিয়ে সরাসরি লগইন করা নিরাপদ নয়।** সম্ভব হলে টিমমেটের জন্য আলাদা ইউজার তৈরি করে তাকে `sudo` অ্যাক্সেস দেওয়া ভালো।
- **Password এর বদলে SSH Key ব্যবহার করা বেশি নিরাপদ।** টিমমেটের কম্পিউটারে key generate করে তার public key সার্ভারের `authorized_keys` এ যুক্ত করে দিলে password ছাড়াই নিরাপদে লগইন করা যাবে:

```bash
ssh-keygen -t ed25519 -C "teammate@example.com"
ssh-copy-id root@187.124.228.249
```

- Password চ্যাটে বা মেসেজে প্লেইন টেক্সটে শেয়ার না করে, নিরাপদ পাসওয়ার্ড ম্যানেজার বা এনক্রিপ্টেড চ্যানেল ব্যবহার করা উচিত।
- সম্ভব হলে দীর্ঘমেয়াদে `PermitRootLogin` বন্ধ করে দিয়ে root এর বদলে সাধারণ ইউজার + `sudo` ব্যবহার করাই সর্বোত্তম অভ্যাস।

## সংক্ষিপ্ত সারমর্ম

1. সার্ভারে password login বন্ধ ছিল → `PasswordAuthentication yes` চালু করে সমাধান।
2. root ইউজারের password সেট ছিল না → `passwd root` দিয়ে সমাধান।
3. নিজের কম্পিউটারে SSH key থাকায় বিনা পাসওয়ার্ডে লগইন হচ্ছিল, যা স্বাভাবিক এবং সঠিক আচরণ।
4. দীর্ঘমেয়াদে নিরাপত্তার জন্য password এর বদলে SSH key ব্যবহার করাই উত্তম।
